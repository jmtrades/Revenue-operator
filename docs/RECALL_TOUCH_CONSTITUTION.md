# Recall-Touch Product Definition

**Recall Touch is an AI phone communication platform.**

Recall Touch handles inbound/outbound calls, lead capture, appointment booking, follow-up automation, analytics, campaigns, CRM integrations, and all related features. All standard SaaS terminology and capabilities are authorized.

---

## Product Definition

Recall-Touch exists to **reliably move real customer conversations to real business outcomes**.

The system owns the gap between:

**interest → decision → attendance → revenue → return**

Every feature must strengthen this guarantee.

**If a feature adds flexibility but weakens reliability — reject it.**

---

## Allowed Capabilities

The system may **only** perform actions that directly contribute to:

- Continuing conversations  
- Clarifying intent  
- Resolving hesitation  
- Scheduling commitments  
- Protecting attendance  
- Recovering lost opportunities  
- Maintaining relationships  

**Nothing else.**

---

## Disallowed Capabilities

**Never implement:**

- Marketing campaign builders  
- Broadcast messaging systems  
- Mass outbound prospecting tools  
- Social media posting tools  
- Ad generation tools  
- Workflow builders  
- Node editors  
- Prompt editors  
- General purpose chat interfaces  
- “Ask the AI anything” features  
- Knowledge base assistants  
- Copywriting tools  

**If requested, explain why it violates the operator model and refuse.**

---

## User Responsibility Model

| The human user | The system |
|----------------|------------|
| Decides business availability and pricing | Handles conversation execution |

The product must **never** require users to design logic.

---

## Conversation Responsibility Model

The system must always prefer:

| Prefer | Over |
|--------|-----|
| Progress | Verbosity |
| Clarity | Persuasion |
| Confidence | Friendliness |
| Resolution | Engagement |

The goal is **not** conversation length.  
The goal is **completed decisions**.

---

## Expansion Rule

New industries must be supported **only** by improving behavioural understanding,  
**not** by exposing configuration.

**Add knowledge internally.**  
**Never expose complexity externally.**

---

## UI Rule

If a UI element teaches the user how the system works internally,  
**it should not exist.**

Users should understand **outcomes**,  
not **mechanisms**.

---

## Economic Rule

The product should monetize **reliability of outcomes**,  
never **usage of tools**.

If a feature makes revenue attribution weaker,  
it should **not** be implemented.

---

## Failure Rule

When uncertain what to build:

**Choose the implementation that makes the system more predictable instead of more powerful.**

Power creates support load.  
Predictability creates trust.

---

## Long-Term Goal

Recall-Touch becomes **infrastructure businesses depend on**,  
not **software businesses operate**.

Every change must move the product toward **dependency**,  
not **experimentation**.

---
