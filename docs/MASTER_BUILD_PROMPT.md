# REVENUE OPERATOR — MASTER BUILD PROMPT

**Source prompt for Business Operations Infrastructure. Living contract: [SYSTEM_DOCTRINE.md](./SYSTEM_DOCTRINE.md).**

---

You are not improving a SaaS application.

You are transforming this repository into a Business Operations Infrastructure.

The system must behave like electricity:
Businesses do not operate it.
They exist inside it.

Your goal is to evolve the current project into a system that eliminates operational uncertainty inside a company.

The system guarantees that work always reaches an outcome, money never silently leaks, and responsibility never disappears.

This is NOT:

- a CRM  
- a chatbot  
- an analytics dashboard  
- a workflow builder  
- a productivity tool  
- an AI assistant  
- a recommendation engine  
- a notification engine  

This IS:

An operational certainty layer that enforces economic outcomes.

The interface is only a record surface.
Behavior is the product.

---

## CORE PURPOSE

Businesses constantly check systems because they do not trust them.

This system replaces checking.

After implementation, a company should no longer wonder:

• Did someone reply?  
• Did we forget a customer?  
• Did a payment fail?  
• Did a lead disappear?  
• Did a staff member drop something?  
• Did a follow-up never happen?  
• Did a booking no-show get ignored?  
• Did a promise never get completed?  

The system guarantees these questions become unnecessary.

---

## PRIMARY PRODUCT GUARANTEES

1) Work never silently stops  
2) Decisions never remain invisible  
3) Commitments always resolve  
4) Recoverable value is never abandoned  
5) Nothing proceeds outside authority without record  

Everything you build must strengthen at least one of these guarantees.

---

## OPERATIONAL CHAIN (MANDATORY ARCHITECTURE)

All behavior MUST flow through this chain only:

**Signal → State → Decision → Authority → Action → Delivery → Reconciliation → Closure → Integrity → Proof**

You may NOT bypass the chain.

**Each layer responsibilities:**

| Layer | Responsibility |
|-------|----------------|
| Signal | Detect real-world events from providers, humans, or system observations |
| State | Canonical representation of business reality |
| Decision | Determine required outcome, not suggestion |
| Authority | If allowed → proceed. If not allowed → create entry and suspend reliance |
| Action | Execute the determined outcome |
| Delivery | Confirm the action actually occurred |
| Reconciliation | Compare reality vs expectation and repair drift |
| Closure | Ensure responsibility reaches final state |
| Integrity | Verify system correctness |
| Proof | Record that the outcome is demonstrably correct |

---

## WHAT THE SYSTEM MUST ELIMINATE

The system replaces human vigilance.

Remove the need for:

- checking inboxes  
- checking CRMs  
- checking calendars  
- checking payments  
- checking staff  
- checking follow-ups  

Humans act only when authority boundary is crossed.

---

## OPERATIONAL DOMAINS TO COVER

The infrastructure must attach to all surfaces where business value can die.

**Communication:** Inbound messages, missed conversations, unanswered prospects, support threads  

**Scheduling:** Bookings, no-shows, reschedules, unconfirmed appointments  

**Payments:** Failed charges, unsent invoices, incomplete checkout, refund escalation  

**Commitments:** Agreements, deliverables, service completion, deadlines  

**Fulfillment:** Orders, requests, tasks requiring outcome, customer promises  

**Retention:** Churn signals, disengagement, unresolved dissatisfaction  

**Team Actions:** Assigned responsibility, dropped handoffs, unacknowledged tasks  

Every domain must map into the operational chain.

---

## ECONOMIC ENGINES (AUTONOMOUS PROCESSES)

Implement the following autonomous outcome engines. They emit canonical signals and run through the same pipeline.

**Revenue Capture** — Recover missed opportunities  

**Revenue Completion** — Finish unfinished transactions  

**Continuity** — Ensure work progresses to resolution  

**Expansion** — Surface legitimate upsell opportunities when state permits  

**Protection** — Prevent loss, churn, or dispute escalation  

These engines do not send suggestions. They enforce outcomes or create entries.

---

## AUTHORITY RULE (CRITICAL)

Before any action:

IF within authority → execute  
IF outside authority → create entry and suspend reliance  

Entries must appear in the UI. The UI does not control the system. It only reveals boundaries.

---

## UI PRINCIPLES

The interface is a record surface only.

Never add: metrics, charts, activity feeds, engagement loops, optimization suggestions, usage gamification, productivity workflows.

Silence equals success.

The UI only shows: state, entries, authority boundaries, attestations.

---

## DEVELOPMENT RULES

For every new behavior ask:

1. What economic failure does this prevent?  
2. What operational state does this protect?  
3. What record exists if authority fails?  

If a feature increases interaction instead of reducing checking — reject it.

Prefer behavior over visibility.

---

## SUCCESS CONDITION

When removed from a business, operations degrade.

- Customers are missed  
- Payments leak  
- Commitments slip  
- Staff forget responsibilities  

With the system present: the company behaves orderly without supervision.

The user does not operate the system. The system operates the business.

## IMPLEMENTATION INSTRUCTIONS

- Refactor existing modules to conform to the operational chain.  
- Add adapters for communication, scheduling, payment, and fulfillment providers.  
- Ensure all processes emit canonical signals.  
- Implement authority resolution before execution.  
- Guarantee reconciliation repairs drift.  
- Record proof for all completed outcomes.  
- Expose only entries and boundaries in UI.  

Do not simplify the doctrine. Do not reinterpret the product as automation software.

You are building operational infrastructure.
