## SECTION 1 — WHAT PREVIOUS ANSWERS STILL GOT WRONG

**1. The prompts treat self-hosted voice as one continuous build.** Tasks 13-17 tell Cursor to build Pipecat + Telnyx + Kokoro + Canary + Llama all in sequence. Cursor will write Python files for GPU inference services that it cannot test, deploy, or verify. These files will be dead code sitting in the repo. Cursor has no access to RunPod, no GPU to test on, and no way to validate that Kokoro inference actually works. The prompts should have separated "write the code" from "deploy and test" — but they didn't.

**What to do:** Accept that Tasks 13-17 produce scaffolding, not working services. The code must be manually deployed to RunPod and tested against real phone calls. Mark these files as `// UNTESTED — requires GPU deployment and real call verification` at the top of each file.

**2. The blog posts (Task 32) are going to be filler.** Ten 1,500-word articles generated by Cursor will read identically. They will use the same sentence structures, the same transition phrases, and the same conclusions. Google's helpful content classifier will flag them.

**What to do:** Do not publish them as-is. Use them as first drafts. Every article needs manual editing: add a personal anecdote or founder perspective, rewrite the introduction to be unique, insert specific numbers from real customer data when available, remove any sentence that could appear in any other article unchanged.

**3. The comparison pages (Task 31) risk legal issues.** Stating competitor pricing ("Smith.ai charges $3.25-$9.50 per call") without sourcing it creates liability if the numbers are wrong or outdated.

**What to do:** Add "Pricing as of March 2026. Visit [competitor].com for current pricing." disclaimer on every comparison page. Link to the competitor's actual pricing page.

**4. The 50-task structure is too rigid for Cursor.** Cursor will hit context limits, lose track of earlier tasks, make mistakes on later tasks, and potentially break earlier work while doing later work. No AI coding agent maintains perfect quality across 50 sequential tasks in one session.

**What to do:** Run Tasks 1-12 in one session. Verify and commit. Run Tasks 13-20 in a second session. Verify and commit. Run Tasks 21-30 in a third. And so on. Never run more than 12-15 tasks per session.

## SECTION 5 — WHAT IS STILL STRUCTURALLY WEAK

**Concurrent call handling is undefined.** When two calls arrive simultaneously for a workspace with only 2 concurrent call slots (Solo tier), the system has no defined behavior. Does the second call queue? Ring busy? Go to voicemail? This must be specified in the Pipecat server: track active calls per workspace, reject excess calls with a voicemail fallback, log as missed call.

**Campaign daily limit carryover is undefined.** If a campaign targets 500 contacts and the daily outbound limit is 100, the process-sequences cron must: process 100 on day 1, remember where it left off, process the next 100 on day 2, and so on. If the cron simply queries "all pending" and tries to send all 500, it will either exceed the limit or silently drop 400 contacts.

**Workspace deletion has no definition.** What happens when a workspace is deleted? Are phone numbers released? Are Stripe subscriptions cancelled? Are recordings deleted? Is the data actually purged or just soft-deleted? This must be defined before any customer uses the product.

## SECTION 6 — WHAT IS STILL FRAGILE

**Google Calendar token refresh.** If a user connected Google Calendar during onboarding but the OAuth token expires (every 60 days without use), the next booking attempt will silently fail. The booking flow must: check token validity before booking, refresh if expired, and if refresh fails, show "Calendar connection expired. Reconnect in Settings." instead of silently failing.

**Telnyx number purchase failure.** If Telnyx API is down when a user tries to buy a phone number, the current flow returns a 502. But the user has no retry mechanism in the UI. They see an error and don't know what to do. Fix: show "Phone number purchase failed. This is temporary — try again in a few minutes." with a retry button that re-submits the same request.

**SMS delivery to landlines.** If the system sends an SMS follow-up to a landline number, the message will fail silently. The follow-up engine should: detect delivery failure from the Telnyx status callback, mark the contact as "SMS undeliverable", and skip SMS steps in future sequences for that contact — switching to call-only.

## SECTION 12 — REVISED FLOW SAFEGUARDS

**Inbound:** Before processing any call, verify: (a) workspace status is active or trial or trial_expired-grace, (b) workspace has at least one live agent, (c) concurrent call count is below tier limit. If any check fails: voicemail fallback.

**Outbound:** Before executing any campaign step, verify: (a) contact has not opted out since enrollment, (b) contact has not been contacted today (suppression), (c) workspace daily outbound count is below limit, (d) current time is within business hours in the contact's timezone. If any check fails: skip this contact, log reason, continue to next.

**SMS delivery failure:** When Telnyx reports delivery failure via status callback: mark contact as sms_undeliverable. Future sequence steps for this contact: replace SMS steps with call steps automatically.
