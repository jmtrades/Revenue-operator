## SECTION 6 — PRODUCT ARCHITECTURE

### 6.1 Universal Omnichannel Inbox
- Why: Single view of all customer conversations across voice, SMS, email, webchat
- MVP: Voice call logs + SMS threads, unified timeline per contact
- Advanced: Full omnichannel with routing rules, team assignment, SLA tracking, sentiment analysis
- Complexity: Medium (event-driven architecture, channel abstraction layer)
- Monetization: Core feature, included in all plans. Premium: team routing, SLA alerts
- Moat: Conversation history creates switching cost. Every month of data makes the platform harder to leave.

### 6.2 Contact/Entity Memory Graph
- Why: AI must remember past interactions, preferences, history, and context across all touchpoints
- MVP: Contact record with call history, notes, tags, basic timeline
- Advanced: Relationship graph (who referred whom, family members at same practice, connected entities), AI-generated contact summaries, predicted next-best-action
- Complexity: Medium-High (vector storage for semantic memory, graph relationships)
- Monetization: Premium AI memory features on higher tiers
- Moat: Deep. The more conversations processed, the better the AI understands each contact. Impossible to replicate by switching tools.

### 6.3 Event Timeline
- Why: Every interaction (call, SMS, email, booking, no-show, payment) on a single timeline per contact
- MVP: Chronological list of events with basic metadata
- Advanced: AI-summarized timeline, "what happened since last contact" briefing, milestone markers
- Complexity: Low-Medium
- Monetization: Core feature
- Moat: Low standalone, high when combined with memory graph

### 6.4 AI Follow-Up Engine
- Why: This is the core differentiator. Automated, intelligent follow-up across channels based on triggers, timing, and context.
- MVP: Rule-based follow-up sequences (if missed call → SMS in 2 min → email in 1 hour → call next day)
- Advanced: AI-determined optimal follow-up timing, channel selection based on contact preferences, dynamic message personalization, escalation rules
- Complexity: High (requires robust workflow engine, channel orchestration, AI decision layer)
- Monetization: Core feature on Growth+. Advanced AI optimization on Scale+.
- Moat: Very high. The AI learns optimal follow-up patterns from aggregate data.

### 6.5 Workflow Automation Engine
- Why: Users need to define custom business rules beyond templates
- MVP: If/then triggers (new lead → assign to agent → send welcome SMS → book callback)
- Advanced: Visual workflow builder, branching logic, delay nodes, condition evaluators, webhook nodes, AI decision nodes
- Complexity: High
- Monetization: Basic workflows on all plans. Advanced builder on Growth+. Custom AI nodes on Scale+.
- Moat: Medium-High. Custom workflows create deep process dependency.

### 6.6 Campaign/Sequence Engine
- Why: Outbound campaigns for reactivation, review requests, seasonal promotions, dead lead resurrection
- MVP: Simple drip sequences (3-5 step SMS/call sequences with timing)
- Advanced: Multi-channel campaigns, A/B testing, AI-optimized send times, dynamic segmentation, campaign ROI tracking
- Complexity: Medium-High
- Monetization: Included on Growth+. Advanced features on Scale+.
- Moat: Medium. Campaign history and performance data create optimization advantage.

### 6.7 Booking & Scheduling Engine
- Why: The #1 outcome businesses want from answered calls
- MVP: Calendar sync (Google, Outlook), availability checking, AI booking during calls, confirmation SMS
- Advanced: Multi-provider scheduling, buffer time, service-type routing, rescheduling automation, waitlist management
- Complexity: Medium
- Monetization: Core feature. Advanced scheduling on Growth+.
- Moat: Medium. Calendar integration creates switching friction.

### 6.8 No-Show Recovery Engine
- Why: 15-30% of booked appointments no-show. Each recovered no-show = pure incremental revenue.
- MVP: Automated reminder sequence (24h, 2h, 30min before appointment). If missed: "We noticed you couldn't make it. Want to reschedule?"
- Advanced: Predictive no-show scoring, preemptive double-booking, dynamic overbooking based on historical rates, waitlist filling
- Complexity: Medium
- Monetization: Included on Growth+. Predictive features on Scale+.
- Moat: High. No-show prediction models improve with data.

### 6.9 Quote/Proposal Chase Engine
- Why: 60-70% of sent quotes never get followed up on
- MVP: Track sent quotes, automated follow-up sequence ("Just checking in on the estimate we sent Tuesday")
- Advanced: AI-powered objection handling, competitive intelligence ("Did you get other quotes? Let me see if we can adjust"), urgency creation, expiration reminders
- Complexity: Medium
- Monetization: Growth+ feature
- Moat: Medium

### 6.10 Reactivation Engine
- Why: Every business has dormant customers worth 5-20x the cost to acquire a new one
- MVP: Identify contacts with no activity in X months, trigger reactivation campaign (call/SMS)
- Advanced: AI-determined reactivation timing based on service cycle (dental: 6mo, HVAC: seasonal, med spa: treatment intervals), personalized reactivation messaging, win-back offers
- Complexity: Medium
- Monetization: Growth+ feature. AI optimization on Scale+.
- Moat: Very high. Reactivation intelligence improves with vertical data. This is a unique, defensible capability.

### 6.11 CRM Sync Layer
- Why: Businesses won't abandon existing CRM. Must be additive.
- MVP: Webhook push to any CRM. Native connectors for HubSpot, Salesforce, ServiceTitan
- Advanced: Bi-directional sync, field mapping UI, conflict resolution, real-time sync
- Complexity: High (each integration is custom work)
- Monetization: Basic webhook on all plans. Native connectors on Growth+. Custom integrations on Scale+.
- Moat: High. Deep integrations create extreme switching cost.

### 6.12 Analytics/Reporting Layer
- Why: ROI proof is the #1 retention driver
- MVP: Calls answered, appointments booked, messages sent, response time, basic revenue estimation
- Advanced: Revenue attribution, cost-per-lead, lifetime value tracking, cohort analysis, competitive benchmarking, executive reports
- Complexity: Medium-High
- Monetization: Basic analytics on all plans. Advanced reporting on Growth+. Executive dashboards on Scale+.
- Moat: Medium. Benchmarking data across the platform creates unique value.

### 6.13 Admin/Org/Team Permissions
- Why: Multi-user teams need role-based access
- MVP: Owner, admin, member roles. Workspace-level isolation.
- Advanced: Custom roles, permission sets, audit logs, SSO, API key management
- Complexity: Medium
- Monetization: Team features on Growth+. SSO/audit on Scale+.
- Moat: Low-Medium. But SSO/compliance features create enterprise lock-in.

### 6.14 Multi-Location Support
- Why: Franchise and multi-location is the highest-ARPU segment
- MVP: Separate workspace per location with shared admin
- Advanced: Location groups, centralized script management, cross-location analytics, per-location billing, centralized vs local control toggles
- Complexity: High
- Monetization: Per-location pricing
- Moat: Very high. Multi-location deployments are extremely sticky.

### 6.15 Agency/Reseller Management
- Why: Distribution leverage
- MVP: Sub-account creation, white-label basics, margin reporting
- Advanced: Full white-label (custom domain, branding, login), agency dashboard, bulk operations, commission tracking, client onboarding templates
- Complexity: High
- Monetization: Agency pricing ($99-$199/sub-account)
- Moat: High. Agency relationships are long-term and create distribution moat.

## SECTION 8 — HIGH-TICKET SALES SYSTEM

### Why This Vertical Is Attractive
High-ticket sales (coaching, consulting, B2B services, financial services, real estate investment) involves $3K-$100K+ deal values. A single recovered no-show or resurrected dead lead can be worth $5K-$50K. Willingness to pay for execution tools is extremely high because the ROI is immediately obvious.

The current tooling landscape for high-ticket sales is fragmented: CRMs that don't act, dialers that don't think, and "sales engagement" platforms built for enterprise B2B, not high-ticket service sales. There's a massive gap.

### Product Design

**Setter Workspace:**
- Lead priority queue (AI-scored: engagement level, intent signals, recency, deal size)
- One-click call with pre-call intelligence briefing ("This lead downloaded your guide 3 days ago, visited pricing twice, company has 50 employees")
- Script/objection guidance (dynamic, not static scripts)
- Disposition logging (qualified → booked, not qualified, callback, no answer)
- Automated follow-up triggering based on disposition
- Show-rate tracking for appointments set
- Performance dashboard: calls made, contacts reached, appointments set, show rate

**Closer Workspace:**
- Upcoming appointments with pre-call intelligence
- Full history/context from setter interactions
- Deal tracking (presented → thinking → closed → lost)
- Post-call follow-up automation (proposal sent → follow-up cadence)
- Deal-stall detection ("This $15K deal has been in 'thinking' for 12 days with no activity — trigger re-engagement")
- Win/loss analysis
- Cash collected tracking

**Manager Workspace:**
- Team performance dashboard (calls, contacts, sets, shows, closes, revenue)
- Rep scorecards and rankings
- Call recordings with AI quality scoring
- Pipeline health (velocity, conversion rates at each stage, stalled deals)

- Typical affiliate brings 1-5 customers per month.
- Your effective CAC: $0-100 depending on affiliate quality.
- Best for: All segments (lowest real CAC).
- Worst for: Nobody (if program is set up well).

**Viral/Referral:**
- Cost: 1 month free per referred customer (if they convert).
- Assume $50/month average plan.
- Cost per referral: $50 (month free).
- But referrals convert at 20-30% (warm intro from friend).
- Effective CAC: $50 / 0.25 = $200.
- BUT: Viral coefficient 0.3 means 30% of customers bring 1 new customer, so 30% of revenue is "free."
- Over time, viral reduces effective CAC by 30%.
- Best for: Solo (viral is natural for solopreneurs).
- Worst for: Enterprise (no viral loops).

**Blended CAC (realistic mix for early stage):**
- Assume: 20% content, 20% partnerships, 20% social, 20% referral, 20% outbound.
- CAC: (20% × $10) + (20% × $60) + (20% × $50) + (20% × $200) + (20% × $2,857) = $634.
- This is high. Reality: Early stage will rely more on content, partnerships, referral (cheaper). Blended CAC $80-150.

### LTV Logic (with Formulas)

**LTV = (ARPU × Gross Margin) / Monthly Churn Rate**

**Or more precisely:**
**LTV = ARPU × GM × (1 / Monthly Churn Rate)**

Where:
- ARPU = Average Revenue Per User per month.
- GM = Gross Margin %.
- Monthly Churn = % of customers lost per month.

**By segment:**

| Segment | ARPU | GM | Churn | LTV |
|---|---|---|---|---|
| Solo | $49 | 65% | 8% | $49 × 0.65 / 0.08 = $397 |
| Starter | $149 | 52% | 5% | $149 × 0.52 / 0.05 = $1,549 |
| Growth | $550 | 45% | 3% | $550 × 0.45 / 0.03 = $8,250 |
| Scale | $1,500 | 50% | 2% | $1,500 × 0.50 / 0.02 = $37,500 |
| Enterprise | $5,000 | 55% | 1% | $5,000 × 0.55 / 0.01 = $275,000 |

**LTV:CAC ratio (measure of efficiency):**
- Solo: $397 LTV / $100 CAC = 3.97x. Good (>3x is healthy).
- Starter: $1,549 / $500 = 3.1x. Good.
- Growth: $8,250 / $2,000 = 4.1x. Good.
- Scale: $37,500 / $3,000 = 12.5x. Excellent.
- Enterprise: $275,000 / $5,000 = 55x. Exceptional.

**All segments have healthy LTV:CAC ratios.** This means acquisition is worthwhile even at early stage.

**LTV expansion scenarios (if you improve product):**

If you reduce churn by 1% annually in each segment:
- Solo: Churn 8% → 7% → LTV $397 → $453 (14% uplift).
- Starter: Churn 5% → 4% → LTV $1,549 → $1,936 (25% uplift).
- Growth: Churn 3% → 2% → LTV $8,250 → $12,375 (50% uplift).
- Scale: Churn 2% → 1% → LTV $37,500 → $75,000 (100% uplift).

**Takeaway:** Reducing churn is the highest-leverage way to increase LTV and profitability.

### Expansion Revenue Logic (Upsell, Cross-Sell, Feature Upgrades)

**Expansion revenue = % of customers upgrading + average increase in ARPU.**

**Targets by segment:**

**Solo → Starter:**
- % of customers upgrading per year: 15%.
- Average Solo customer on 1 year: 50% still around (after churn).
- Of those 50%, 15% upgrade to Starter: 50% × 15% = 7.5% of initial cohort.
- Average plan upgrade: $49 → $149 = +$100/month = +$1,200/year per upgrading customer.
- Expansion revenue per customer: 7.5% × $1,200 = $90/year = $7.50/month.

**Starter → Growth:**
- % upgrading per year: 10%.
- % retained: 54%.
- % of cohort upgrading: 54% × 10% = 5.4%.
- Upgrade: $149 → $550 = +$401/month = +$4,812/year.
- Expansion revenue per customer: 5.4% × $4,812 = $260/year = $21.70/month.

**Growth → Scale:**
- % upgrading per year: 5%.
- % retained: 66%.
- % of cohort: 66% × 5% = 3.3%.
- Upgrade: $550 → $1,500 = +$950/month = +$11,400/year.
- Expansion revenue per customer: 3.3% × $11,400 = $376/year = $31/month.

**Net Revenue Retention (NRR) = (Beginning MRR + Expansion - Churn) / Beginning MRR.**

If you have:
- Beginning MRR: $10,000.
- Expansion MRR: $700 (expansion from existing customers).
- Churn MRR: $500.
- Ending MRR: $10,000 + $700 - $500 = $10,200.
- NRR: $10,200 / $10,000 = 102%.

**NRR > 100% means:** You're growing faster via expansion than you're shrinking via churn. This is the sign of a healthy SaaS business.

**Target NRR:** 110-120% (you're growing from expansion + new customers combined faster than churn).

**Expansion tactics:**
1. **Feature-based upsells:** "You've used workflows 200 times. Upgrade to Growth to unlock A/B testing and advanced reporting."
2. **Seat expansion:** "You've invited 3 team members but your plan only allows 2. Upgrade for 5 seats."
3. **Usage upsells:** "You're at 80% of your minute limit. Upgrade to Growth for more minutes and better analytics."
4. **Vertical expansion:** "You're doing invoice collection. Want to add callback management?" (new module, new price).

## SECTION 18 — TRUST, RISK, AND COMPLIANCE

### Risk: AI Hallucination / Misinformation
- Danger: AI tells a caller the wrong business hours, quotes a wrong price, makes a false promise
- Mitigation: Strict grounding — AI only uses information explicitly provided in the knowledge base. If uncertain, it says "Let me have someone get back to you on that." Configurable "never say" rules. Call recording review.
- Product safeguard: "Guardrails" settings page where businesses define what AI can/cannot say. Post-call AI audit flagging potential issues.

### Risk: Wrong Follow-Up Timing
- Danger: Sending a follow-up at 2am, following up too aggressively, following up on a cancelled request
- Mitigation: Business hours enforcement on all outbound. Contact preference tracking. Unsubscribe/opt-out handling. Frequency caps.

### Risk: Bad Automation Behavior
- Danger: Infinite loop of follow-ups, sending to wrong contact, duplicate messages
- Mitigation: De-duplication logic, maximum message limits per contact per period, human review queue for edge cases, "pause all automation" emergency button.

### Risk: Spam/Telecom Compliance
- Danger: Getting phone numbers flagged as spam, violating TCPA, A2P messaging regulations
- Mitigation: Opt-in tracking, proper A2P 10DLC registration, call frequency limits, "STOP" handling for SMS, caller ID reputation monitoring, compliance documentation.

### Risk: Privacy & Data Security
- Danger: Call recordings containing sensitive information, HIPAA violations for healthcare, GDPR for EU
- Mitigation: Encryption at rest and in transit, role-based access, data retention policies, HIPAA BAA for healthcare tier, call recording consent announcements, data export/deletion on request.

### Risk: Bad AI Voice Experience
- Danger: AI sounds robotic, has latency, misunderstands the caller, creates frustrating experience
- Mitigation: Invest in top-tier voice synthesis (ElevenLabs Conversational), tune latency aggressively (<800ms response time target), implement graceful fallback ("I didn't catch that, could you repeat?"), offer call transfer to human as escape hatch.

### Risk: Low Trust Onboarding
- Danger: Business owners don't trust AI to talk to their customers
- Mitigation: Test call during onboarding (hear the AI yourself), call recording access from day 1, "review mode" where AI answers but sends summary to owner before taking action, gradual trust-building (start with after-hours only, expand to all calls).

### Trust-Building UX Elements
- Transparent AI: "Your AI said this on the call" with full transcript
- Control: Easy override of any AI action, "teach the AI" feedback mechanism
- Proof: "Here's what would have happened without Recall Touch" counterfactual
- Safety: One-click pause all AI activity, instant human takeover

## SECTION 19 — WHAT TO CUT / WHAT NOT TO BUILD

### CUT or DELAY:

**1. Full CRM** — Do not build a CRM. Integrate with existing ones. Building a CRM is a 5-year, $50M+ project and you will lose to HubSpot, Salesforce, and ServiceTitan. Stay in the execution layer.

**2. Email marketing platform** — Do not build Mailchimp. Use email as a channel within follow-up sequences, but don't build campaign design tools, template builders, or email analytics dashboards.

**3. Social media management** — Completely out of scope. Do not touch this.

**4. Website builder / landing pages** — Not your business. Refer to partners.

**5. Generic chatbot for websites** — Commoditized, low value, distracts from voice/phone which is the hero capability.

**6. Video calling / telehealth** — Different product category entirely. Don't get distracted.

**7. Inventory / POS systems** — Vertical-specific infrastructure. Not your problem.

**8. Accounting / invoicing (full)** — Don't build QuickBooks. Invoice CHASING (follow-up on overdue invoices) yes. Invoice CREATION no.

**9. Solo user as primary target** — Build it eventually but never prioritize over business users. The economics don't work as a primary segment.

**10. Free tier** — Never. Free trial yes. Free ongoing no. Free users consume support, create noise, and never convert at rates that justify the cost.

**11. AI image generation, document creation, or general AI assistant features** — Stay focused. You are a revenue execution system, not a general AI tool.

**12. WhatsApp Business integration (early)** — Important eventually but complex to implement (Meta approval process, template restrictions). Delay until after voice + SMS are proven.

**13. Mobile app (native, early)** — Responsive web app first. Native app only when there are 1,000+ daily active users demanding it. Mobile web with push notifications covers 90% of needs.

**14. Marketplace / app store** — Way too early. Don't build extensibility infrastructure until you have 5,000+ customers.

### Features That Dilute Positioning
- "AI writing assistant" — makes you look like a chatGPT wrapper
- "Project management" — makes you look like a generic tool
- "Team chat / internal messaging" — that's Slack's job
- "Document storage / file sharing" — that's Google Drive's job

### Expensive Infrastructure Traps
- Real-time voice transcription display during calls (impressive demo, marginal value, very expensive)
- Custom voice cloning (legal/ethical minefield, high cost, low retention impact)
- Multi-language AI voice (important eventually, but don't do more than English + Spanish in year 1)

## 3. BACKEND / SYSTEM REQUIREMENTS

### Workflow Engine
- Event-driven architecture: triggers fire on events (missed_call, new_lead, no_show, quote_sent, appointment_created, contact_dormant)

- Channel orchestrator: routes execution to correct channel handler (Twilio for voice/SMS, SendGrid for email)
- Enrollment manager: tracks contact position in sequences, handles deduplication, opt-outs, max frequency

### Scheduling Engine
- Calendar sync: Google Calendar + Outlook bi-directional sync via OAuth
- Availability calculator: merge business hours + calendar events + buffer time → available slots
- Booking handler: create appointment, sync to calendar, send confirmation SMS, schedule reminders
- Reminder chain: 24h → 2h → 30min before appointment, configurable per workspace

### Event Processing
- Real-time event bus for: call started, call ended, SMS received, appointment created, lead captured, follow-up due
- Webhook delivery system for CRM sync and external integrations
- Event replay capability for debugging
- Dead letter queue for failed deliveries

### Conversation Memory
- Per-contact memory store: key facts, preferences, history summary (stored as structured JSON + optional vector embedding)
- AI-generated contact summaries after each interaction
- Memory retrieval during live calls (feed context to AI agent)

### Lead Scoring / Priority Layer
- Rule-based initial scoring: recency, engagement level, deal value, source quality
- ML-based scoring later: predict conversion probability based on historical patterns
- Priority queue for follow-up: highest-scored leads get followed up first

### Analytics Aggregation
- Daily rollup job: compute daily_metrics from raw events
- Revenue estimation: map call outcomes to estimated revenue based on industry benchmarks
- Real-time counters for dashboard: calls today, appointments today, using Redis/in-memory cache

### Billing Meter Events
- Track every billable event: voice minutes, SMS sent, outbound calls, contacts contacted
- Real-time usage counter accessible via API for dashboard display
- Threshold alerts at 80% and 100% of plan limits
- Overage calculation and Stripe invoice item creation

### Role-Based Access Control
- Workspace-level roles: owner, admin, member
- Permission matrix: who can view calls, edit agents, manage billing, invite members, access API
- Audit log for sensitive actions (settings changes, agent modifications, billing changes)

### Observability
- Structured logging for all API routes and background jobs
- Error tracking with context (workspace_id, user_id, action)
- Performance metrics: API response times, voice latency, job queue depth
- Cost tracking: per-workspace cost attribution (voice minutes × rate + SMS × rate)

- Duration: 14 days for new accounts, all features at Pro level

### Stripe Implementation Specifics
- Products: one Stripe Product per plan tier (solo, starter, growth, pro, enterprise)
- Prices: recurring Price objects, monthly and annual billing cycles
- Subscriptions: subscription object created with plan Price, stores Stripe subscription ID in DB
- Payment Method: Stripe PaymentMethod attached to Customer on signup
- Billing Portal: embed Stripe Customer Portal in dashboard (plan changes, payment method, invoices, cancel)
- Webhooks: listen to invoice.payment_succeeded, invoice.payment_failed, customer.subscription.updated, customer.subscription.deleted
- Retries: Stripe handles automatic retry on payment failure (default 3 attempts over 2 days)
- Dunning: after final failed payment, subscription paused, email with update payment link
- Tax: use Stripe Tax for VAT/GST calculation, store tax_id on Customer
- Invoices: auto-generated by Stripe, forwarded to customer email, also visible in portal
- Seat upgrades: create immediate invoice for prorated difference (Stripe uses metered billing for this)
- Usage metering: use Stripe Billing meter_event for voice minutes, SMS, to track month-to-date overage

### Cost Guardrails
- Monthly spend cap: customer can set max monthly invoice amount ($500-$10000)
- Alert thresholds: email alerts at 50%, 75%, 90%, 100% of cap
- Hard limit at 110%: system throttles inbound calls/SMS if spending hits 110% of cap (protect from runaway overage)
- Auto-pause option: customer can opt to auto-pause account if cap exceeded, resume with approval
- Overage pre-approval: workflows cannot be launched if estimated cost exceeds cap
- Annual vs monthly: spend cap applies monthly, annual subscriptions don't have overage (capped minutes)

### Upgrade / Downgrade Flows
- **Upgrade**: immediate, prorated charge for remaining days at new plan rate, added to next invoice

- **Downgrade**: takes effect on next billing cycle, no immediate charge
  - Warning: "Downgrade takes effect on April 1. You'll lose access to: [features]"
- **Mid-cycle changes**: calculated by Stripe automatically if using proration
- **Cancellation**: effective immediately, customer retains access until end of current billing period
  - Final invoice: prorated credit if upgrading to lower tier, charge if upgrading to higher
- **Reactivation**: if cancelled within 30 days, restore settings and campaigns, re-enable subscription
- **Plan comparison**: modal showing current plan vs. target plan, highlighting gains/losses, estimated cost change

## 6. ANALYTICS / KPI REQUIREMENTS

### Response Speed Metrics
- **Avg Response Time**: seconds from contact message to first reply (SMS/call)
- **Response Time by Channel**: inbound SMS, inbound call, outbound follow-up
- **First Response SLA**: % of contacts responded to within 2 hours (configurable threshold)
- **Response Rate by Time of Day**: response speed at 9am, 12pm, 3pm, 6pm to identify slow periods
- **Agent Response Comparison**: leaderboard of agents by avg response time
- **Trend**: week-over-week comparison with arrow indicator

### Appointments Booked Metrics
- **Total Appointments**: count of confirmed appointments (not cancellations)
- **Appointment Rate**: % of contacts with an appointment booked (contacts with appts / total contacts)
- **Appointment Value**: total estimated revenue from booked appointments
- **Booking by Channel**: % booked via SMS, call, workflow automation
- **Booking by Agent**: leaderboard of top agents by appointments booked
- **Booking by Campaign**: which campaigns drive most appointments
- **Show Rate**: % of appointments that were attended (opposite of no-show)
- **Trend**: appointments this month vs. last month, with % change

### No-Show Recovery Metrics
- **No-Shows**: count of missed appointments (contact didn't show)
- **No-Show Rate**: no-shows / total appointments %
- **Recovered No-Shows**: count of re-booked appointments from no-show follow-up
- **Recovery Rate**: recovered / total no-shows %
- **Recovery Revenue**: estimated revenue from recovered appointments
- **Recovery Time to Rebook**: average days between original missed appt and rebooked appt
- **Recovery by Agent**: which agents are best at re-booking no-shows
- **Outcome of Recovery Attempts**: attended, another no-show, declined to rebook

### Reactivation Wins Metrics
- **Inactive Contacts**: count of contacts with no activity in 90+ days
- **Reactivation Campaigns Sent**: count of re-engagement workflow launches
- **Reactivated Contacts**: contacts that responded after reactivation campaign
- **Reactivation Rate**: reactivated / contacted %
- **Reactivation Revenue**: estimated value from reactivated contacts
- **Reactivation by Lifecycle Stage**: which stages respond best (cold leads, past customers, former agents)
- **Top Reactivation Sequences**: which workflow templates drive most responses
- **Time to Response**: days from campaign send to first response

### Revenue Influenced / Recovered Metrics
- **Total Revenue Influenced**: sum of appointment values + estimated closed deals from Recall Assist contacts
- **Revenue Influenced by Campaign**: per-campaign revenue attribution
- **Revenue Influenced by Agent**: per-agent contribution to deals
- **Revenue Influenced by Channel**: phone vs SMS vs email effectiveness
- **Deal Closed**: count of contacts that became paying customers (requires CRM integration)
- **Deal Size**: average deal value from Recall Assist contacts
- **Sales Cycle Time**: days from first contact to deal closed
- **Win Rate**: contacts that booked → contacts that paid %
- **Cost Per Win**: total Recall Assist cost / number of deals closed
- **Pipeline Value**: sum of all open opportunities from Recall Assist contacts

### Pipeline Progression Metrics
- **Pipeline Stage Breakdown**: count of contacts at each stage (lead, qualified, proposal, negotiation, closed)
- **Stage Duration**: average days spent in each stage
- **Stage Conversion**: % of contacts advancing to next stage
- **Bottleneck Identification**: stages with longest duration or lowest conversion rate
- **Velocity**: contacts progressing per week by stage
- **Flow Diagram**: Sankey chart showing stage transitions and drop-off
- **Forecast**: projected closures in next 30/60/90 days based on current pipeline
- **At-Risk Contacts**: contacts stalled in a stage for >7 days

### Usage Metrics
- **Voice Minutes Used**: total monthly, trending, % of plan limit
- **SMS Sent**: total monthly, trending, % of plan limit
- **Outbound Calls Made**: total monthly, trending, % of plan limit
- **API Calls**: total monthly (for paid API tier customers)
- **Contacts Managed**: total active contacts in CRM
- **Workflows Executed**: total workflow runs, average per workflow template
- **Campaigns Active**: count of active campaigns
- **Agent Activity**: logins, calls handled, SMS sent per agent per day
- **Peak Usage Times**: hours/days with highest usage to identify congestion
- **Usage Forecast**: projected month-end usage based on current burn rate

### Churn-Risk Metrics
- **Usage Decline**: customers with >20% drop in usage month-over-month (signal of disengagement)
- **Payment Issues**: customers with failed payments in last 30 days
- **No Login**: customers with no user login in 30+ days
- **Segment Performance**: churn rate by plan tier, by industry, by age cohort
- **Expansion Indicator**: customers increasing usage (lower churn risk)
- **Support Tickets**: customers with 3+ support tickets in month (may indicate friction)
- **NPS Score**: sent quarterly, <6 = detractor (likely to churn)
- **Churn Risk Score**: ML model combining above signals, 0-100 scale
- **At-Risk List**: customers scoring 70+, grouped by reason (payment, usage, support)

### Cost-to-Serve Metrics
- **COGS per Contact**: (voice + SMS + API cost) / contacts managed
- **COGS per Minute**: total infrastructure cost / total voice minutes
- **Gross Margin**: (ARR - COGS) / ARR %
- **Margin by Plan**: gross margin per tier (to identify unprofitable plans)
- **Customer Acquisition Cost (CAC)**: total sales & marketing spend / new customers
- **CAC Payback Period**: months to recover CAC from customer MRR
- **Lifetime Value (LTV)**: projected revenue per customer over lifetime
- **LTV/CAC Ratio**: should be 3:1 or higher (healthy business model)
- **Burn Rate**: monthly infrastructure cost without revenue offset
- **Profitability by Segment**: which use case, industry, size customer is most profitable

### Agent Performance Metrics
- **Calls Handled**: inbound calls answered by agent
- **Call Duration**: average call length
- **Call Conversion**: % of calls that resulted in appointment/action
- **SMS Response Rate**: % of SMS messages responded to by agent
- **First Response Time**: average seconds to first reply from agent
- **Quality Score**: if call recording/transcription enabled, AI-scored call quality 0-100
- **Appointment Book Rate**: % of agent's calls that resulted in appointment
- **Escalation Rate**: % of calls escalated to supervisor/admin
- **After-Call Work**: average time spent on notes/CRM updates post-call
- **Schedule Adherence**: % of time agent was available vs. scheduled

## 7. INFRASTRUCTURE / COST-CONTROL REQUIREMENTS

### Usage Metering Architecture
- **Metering Service**: dedicated service that tracks usage events (call started, SMS sent, API request)
- **Event Types**: voice_call_started, voice_call_ended, sms_sent, outbound_call_started, outbound_call_ended, api_request, workflow_executed
- **Real-time Buffer**: batch events in-memory for 5 seconds or 100 events, then flush to database
- **Per-Tenant Tracking**: meter_usage table has (account_id, event_type, count, timestamp, period_month)
- **Idempotency**: events have unique ID, deduplicate on flush to prevent double-counting
- **Accuracy**: meter_usage updated atomically with billing system for monthly invoice generation
- **Backpressure**: if metering service falls behind, buffer up to 10,000 events in Redis, warn on logs
- **Reconciliation**: daily job compares billed usage vs. raw event logs, alerts on >5% discrepancy

### Per-Feature Cost Attribution
- **Voice Inbound**: $0.007 per minute (carrier cost + infrastructure)
- **Voice Outbound**: $0.012 per minute (2x inbound due to outbound call cost)
- **SMS Inbound**: $0.001 per SMS (carrier cost, minimal)
- **SMS Outbound**: $0.0075 per SMS (carrier cost + platform)
- **API Request**: $0.00001 per request (negligible, included in plans)
- **Workflow Execution**: $0.001 per workflow run (compute cost amortized)
- **Storage**: $0.05 per GB-month for call recordings, $0.02 per GB-month for contact data
- **Transcription**: $0.005 per minute of call transcribed (if enabled)
- **Use cost tracking**: cost_breakdown table with (account_id, feature, count, unit_cost, total_cost, date)
- **Margin calculation**: selling_price - total_cost = gross margin, tracked daily per account

### Caching Strategy
- **Redis Cache Layer**: all contacts, campaigns, workflows cached in Redis with 1-hour TTL
- **Cache Keys**: contact:{account_id}:{contact_id}, campaign:{account_id}:{campaign_id}, workflow:{account_id}:{workflow_id}
- **Invalidation**: on contact/campaign/workflow update, invalidate cache key immediately + republish to Redis pub/sub
- **Session Cache**: user session data (account_id, role, preferences) cached in Redis for 8 hours
- **Rate Limit Cache**: track API calls per account in Redis for rate limiting logic
- **Analytics Cache**: pre-calculated KPI values cached for 1 hour, updated on schedule job
- **Cache Warming**: on app startup, warm cache with top 100 active accounts' contacts + campaigns
- **Cache Fallback**: if Redis unavailable, read from database (slower, but functional)
- **Cache Size Monitoring**: alert if Redis memory usage >80%, evict oldest entries using LRU

### Batching & Async Processing
- **Outbound SMS Batch**: queue outbound SMS requests, batch 100 at a time, submit to Twilio API every 10 seconds
  - Benefit: reduced API calls, Twilio charges per request, batching saves ~50% API cost
- **Call Recording Batch**: after call ends, queue recording for async processing (transcription, archival)
  - Benefit: don't block user during transcription (slow), process in background
- **Webhook Batch**: queue CRM/Salesforce webhook events, batch 50 per request, send every 5 minutes
  - Benefit: reduce outbound network calls, CRM API rate limits
- **Report Generation**: background job, generate monthly PDF reports in async queue, email to customer
- **Usage Sync**: sync usage meter events to Stripe every hour (don't real-time sync, costs money)
- **Job Queue**: use Bull (Node.js) or Celery (Python) for async job management
- **Job Retry**: failed jobs retry 3x with exponential backoff (1s, 5s, 30s)
- **Job Deadletter**: if 3 retries fail, move to dead letter queue, alert ops
- **Job Timeout**: max 5 minutes per job, kill stalled jobs

### Vendor Abstraction Layer
- **SMS Abstraction**: interface supports Twilio, AWS SNS, Bandwidth
  - Config: customer selects vendor in settings, system routes via adapter pattern
  - Cost: negotiate volume discounts with multiple vendors, route to cheapest option
- **Voice Abstraction**: interface supports Twilio, AWS Connect, Vonage
  - Failover: if primary vendor unavailable, route to secondary vendor automatically
  - Metrics: track cost per vendor, alert if one vendor significantly more expensive
- **Transcription Abstraction**: interfaces support Google Speech-to-Text, AWS Transcribe, Deepgram
  - Quality vs. Cost: customer selects quality tier (fast/cheap vs. high-quality)
  - Caching: cache transcription results in database, don't re-transcribe same audio
- **LLM Abstraction**: interface supports OpenAI, Anthropic Claude, Azure OpenAI
  - Model selection: customer selects model (gpt-3.5-turbo = cheap, gpt-4 = expensive)
  - Cost limiting: track LLM spend per account, alert if approaching monthly limit
- **Adapter Pattern**: each vendor is a class implementing common interface, swappable at runtime

### Failure Handling & Fallback
- **Voice Call Failure**: if Twilio call fails, retry with AWS Connect, if that fails, notify user "call temporarily unavailable"
- **SMS Delivery Failure**: if SMS fails, queue for retry next day, if fails 3x, notify customer "SMS not delivered to this number"
- **API Timeout**: if external API call takes >10 seconds, timeout and return cached response (if available)
- **Database Connection Failure**: use connection pooling with reconnect logic, auto-failover to replica if primary unavailable
- **Payment Processing Failure**: Stripe payment fails → send customer email with update payment link, pause account after 3 failed attempts
- **Cascade Limit**: don't have cascading failures (one service down takes down entire app), use circuit breakers
- **Graceful Degradation**: if analytics service down, still allow calls/SMS, just don't show metrics
- **Monitoring**: use Sentry for error tracking, alert on errors >5% of requests

### Channel Fallbacks
- **Primary Inbound**: phone number (Twilio) → if caller hangs up before speech-to-text completes, send SMS reminder "Call dropped, reply to continue"
- **Outbound Strategy**: Email → SMS → Call (if customer has email but not phone, try email first)
  - Configurable: customer can set priority order in settings
- **SMS to Call**: if SMS delivery fails and customer has configured, auto-escalate to call
- **Call to SMS**: if inbound call not answered by agent, auto-send SMS "Call us back: [link to schedule call]"
- **Webhook Fallback**: if webhook delivery fails, store in database, retry every hour for 24 hours
- **Vendor Fallback**: track vendor reliability, auto-switch to secondary vendor if primary >10% error rate

### Cost Alerts & Notifications
- **Daily Cost**: customer receives daily email showing spend to date, "on track" vs. overage
- **Usage Alert**: when account reaches 80% of monthly limit, email warning + in-app notification
- **Overage Alert**: when account exceeds monthly limit, email alert, daily notification until next month
- **Spend Cap Alert**: when spend cap reached, email alert, workflows blocked until cap reset
- **Cost Spike Detection**: if spend increases >50% month-over-month, email alert "unusual activity detected"
- **Budget Anomaly**: AI model trained on account usage history, alerts if usage pattern unusual (e.g., suddenly 10x calls)
- **Margin Warning**: for resellers/agencies, alert if sub-account cost >85% of selling price (margin erosion warning)
- **Underutilization**: account using <20% of plan, email suggestion to downgrade to cheaper plan (help them self-optimize)

### Margin Protection
- **Dynamic Pricing**: enterprise customers get custom pricing based on volume + margin target (never sell below 40% margin)
- **Usage Cap**: overages hard-capped at 125% of plan included minutes (prevent runaway costs)
- **Auto-Scale Limits**: limit concurrent calls to (account_plan_tier * 10), queue excess (don't burn cash on infrastructure overage)
- **Cost Review**: monthly business review with top 20 customers, discuss unit economics, identify optimization opportunities
- **Vendor Optimization**: quarterly review of vendor costs, negotiate volume discounts, switch vendors if >10% savings possible
- **Resource Allocation**: infrastructure resources allocated by account margin (high-margin = better resource priority)
- **Profitability Scoring**: every account scored 0-100 on profitability (margin, LTV, CAC), resource decisions reflect score
- **Sunset Unprofitable**: if account margin <5%, offer buyout incentive to migrate to enterprise license (one-time payment)

- **Motion**:
  - Button press: scale down 2% on active, spring-like return
  - Expand/collapse: max-height transition with 200ms ease
  - Notification slide: slide-in from top-right, 300ms ease
  - Fade in: opacity 0 → 1, 300ms ease
- **Avoid**: jarring animations, motion on scroll (accessibility)
- **Reduced Motion**: respect prefers-reduced-motion, disable animations for users

### Empty / Loading / Error / Success States

**Empty State**:
- Large icon (32-48px) in neutral color (#D1D5DB)
- Heading: "No data yet" or "No [contacts/campaigns/calls]"
- Subheading: "Create your first [contact/campaign/call] to get started"
- CTA button: "Create [Item]" or "Get Started"
- Illustration or icon-based visual (not photo)
- Center-aligned, takes up ~50% of container width

**Loading State**:
- Skeleton loaders for known layouts (contact rows, KPI cards)
  - Gray placeholder bars, same height/width as real content
  - Shimmer animation (left-to-right sweep) at 1.5s duration
- Full-page load: spinner icon (Heroicons LoadingSpinner) in center of page
  - Size: 32px, color #2563EB, smooth rotation animation
  - Subtitle: "Loading..." or specific text ("Fetching contacts...")
- Table loading: show 5 skeleton rows, full width

**Error State**:
- Alert card: red background, error icon, heading "Something went wrong"
- Error message: specific error from API, truncated if >100 chars
- Action buttons:
  - "Retry" button (primary)
  - "Go Back" link (secondary)
- Optional: error code/ID for support reference
- Sentry integration: error auto-reported with user context

**Success State**:
- Toast notification: slide in from top-right, green background, icon + message
  - Auto-dismiss after 4 seconds
  - Example: "Call recorded successfully"
- Inline success: green checkmark icon + "Saved" message in form
- Page-level success: success alert with action confirmation
  - Example: "Campaign launched! View results"
  - CTA button: "View Campaign" link
